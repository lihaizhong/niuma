## Context

当前 Niuma 项目已完成核心基础设施、Agent 核心和多个内置工具（文件系统、Shell、Web、Git、压缩解压、网络、数据处理）。项目使用 TypeScript 严格模式，所有工具都继承自 `SimpleTool` 基类，并注册到工具注册表中。

Node.js 内置的 `crypto` 模块提供了完整的加密功能，包括：
- 对称加密（AES）
- 哈希算法（SHA、MD5）
- 随机数生成
- 密钥派生

这些功能无需额外依赖，性能优异，适合作为内置工具实现。

## Goals / Non-Goals

**Goals:**
- 提供安全可靠的数据加密/解密功能（AES-256-GCM）
- 提供多种哈希算法（SHA-256、SHA-512、MD5）
- 遵循现有工具模式，保持代码一致性
- 完整的类型安全和错误处理
- 100% 测试覆盖率

**Non-Goals:**
- 非对称加密（RSA、ECC）- 如有需要可通过 MCP 对接
- 密钥管理（KMS、Vault）- 超出范围
- 数字签名 - 非必需功能
- 性能优化（使用 WASM 或 GPU 加速）- 非必需

## Decisions

### 1. 加密算法选择：AES-256-GCM

**决策：** 使用 AES-256-GCM 对称加密算法

**理由：**
- **安全性**：GCM 模式提供认证加密（AEAD），同时保证机密性和完整性
- **性能**：现代 CPU 硬件加速（AES-NI），性能优异
- **标准性**：NIST 推荐标准，广泛应用
- **IV 安全**：每次加密使用随机 IV，避免 IV 重用攻击

**替代方案考虑：**
- AES-256-CBC：不提供认证，需要额外 HMAC，复杂度高
- ChaCha20-Poly1305：安全性相当，但 AES-256-GCM 硬件支持更广泛

### 2. 密钥管理策略

**决策：** 工具接受密钥作为参数，不存储密钥

**理由：**
- **安全性**：避免密钥泄露风险
- **灵活性**：用户可自行管理密钥（环境变量、密钥管理服务等）
- **责任分离**：Niuma 不承担密钥管理责任

**实现方式：**
```typescript
// encrypt 工具参数
{
  data: string;      // 要加密的数据
  key: string;       // 密钥（Base64 编码）
  iv?: string;       // 可选 IV（Base64 编码），如未提供则生成随机 IV
}

// 返回值
{
  encrypted: string;  // 加密后的数据（Base64 编码）
  iv: string;         // 使用的 IV（Base64 编码）
  tag: string;        // 认证标签（Base64 编码）
}
```

### 3. 哈希算法实现

**决策：** 支持 SHA-256、SHA-512 和 MD5

**理由：**
- **SHA-256**：通用性强，性能和安全性平衡
- **SHA-512**：更高安全性，适合敏感场景
- **MD5**：兼容性需求（虽然安全性不足，但在某些场景仍有用途）

**实现方式：**
```typescript
// hash 工具参数
{
  data: string;      // 要计算哈希的数据
  algorithm: 'sha256' | 'sha512' | 'md5';
}

// 返回值
{
  hash: string;      // 哈希值（十六进制字符串）
}
```

### 4. 错误处理策略

**决策：** 明确的错误类型和错误消息

**理由：**
- **调试友好**：清晰的错误信息帮助用户快速定位问题
- **类型安全**：使用 TypeScript 类型系统确保错误处理
- **国际化**：错误消息使用中文，符合项目规范

**错误类型：**
- `CryptoError`：加密相关错误（密钥格式错误、加密失败等）
- `ValidationError`：参数验证错误（缺少必需参数、算法不支持等）

### 5. 测试策略

**决策：** 100% 单元测试覆盖，包括边界情况

**理由：**
- **安全性关键**：加密功能错误可能导致数据泄露
- **质量保证**：确保所有代码路径都经过验证
- **回归测试**：防止未来修改引入问题

**测试覆盖：**
- 正常情况：加密、解密、哈希
- 边界情况：空字符串、大文件、特殊字符
- 错误情况：错误密钥格式、错误算法、解密失败
- 安全性：密钥长度验证、IV 唯一性

## Risks / Trade-offs

### 风险 1：密钥泄露
**风险：** 用户可能在日志中暴露密钥
**缓解措施：**
- 工具返回值中不包含密钥
- 在文档中明确警告不要在日志中记录密钥
- 使用环境变量存储密钥

### 风险 2：密钥管理不当
**风险：** 用户可能使用弱密钥或硬编码密钥
**缓解措施：**
- 在文档中提供密钥生成建议
- 建议使用密钥管理服务
- 提供密钥强度验证（可选）

### 权衡 1：性能 vs 安全性
**权衡：** 更强的安全性（如 PBKDF2 密钥派生）会增加性能开销
**决策：** 不实现密钥派生，保持工具简洁，用户自行管理密钥

### 权衡 2：功能范围
**权衡：** 支持更多算法会增加复杂度
**决策：** 只实现最常用的算法（AES-256-GCM、SHA-256/512、MD5）
