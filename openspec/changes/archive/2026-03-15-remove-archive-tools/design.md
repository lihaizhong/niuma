## Context

当前 Niuma 项目已完成 Phase 3.3 的压缩与解密工具（archive 和 extract），使用 adm-zip、archiver、tar 等依赖。这些工具虽然功能完整，但对本地资源消耗较大：

- **CPU 密集型**：压缩和解压需要大量 CPU 计算
- **内存占用**：处理大文件时需要加载整个文件到内存
- **依赖复杂**：需要维护多个压缩库（adm-zip、archiver、tar）
- **性能瓶颈**：无法利用 GPU 加速等优化

项目正在采用 MCP 优先架构，将资源密集型功能（如图像处理、媒体处理）转移到 MCP Server。Archive 工具应该遵循同样的策略。

## Goals / Non-Goals

**Goals:**
- 删除 archive 和 extract 工具及其依赖
- 减轻 Niuma 核心的资源消耗
- 保持代码库简洁和易维护
- 为 MCP 对接提供清晰路径

**Non-Goals:**
- 不替代为其他内置压缩实现
- 不保留 archive 工具的任何代码
- 不提供内置的替代方案

## Decisions

### 1. 完全删除而非重构

**决策：** 完全删除 archive 工具，而不是优化或重构

**理由：**
- **资源消耗是固有的**：无论怎么优化，压缩/解密本质上是资源密集型操作
- **MCP 优先架构**：项目已决定将此类功能通过 MCP 提供
- **降低维护成本**：删除代码比维护代码更简单
- **避免技术债务**：保留不使用的功能会增加技术债务

**替代方案考虑：**
- 优化现有实现：无法解决根本的资源消耗问题
- 使用更高效的库：仍然是资源密集型，且会增加复杂度

### 2. 删除所有相关依赖

**决策：** 从 package.json 中删除所有压缩相关依赖

**理由：**
- **减少安装时间**：减少依赖包的数量
- **减少安全风险**：减少潜在的依赖漏洞
- **保持精简**：只保留必要的依赖

**受影响的依赖：**
- adm-zip
- archiver
- tar

### 3. 清理测试文件

**决策：** 删除所有 archive 相关的测试文件

**理由：**
- **测试无意义**：删除功能后，测试不再有作用
- **保持一致性**：代码和测试同步删除

### 4. 文档更新

**决策：** 更新开发计划和变更日志

**理由：**
- **准确记录**：反映当前项目状态
- **避免混淆**：防止用户误以为功能仍然可用

## Risks / Trade-offs

### 风险 1：现有用户依赖 archive 工具
**风险：** 可能有用户正在使用 archive 和 extract 工具
**缓解措施：**
- 在 CHANGELOG 中明确标记为 BREAKING CHANGE
- 提供清晰的迁移建议（使用 MCP Server）
- 给出足够的通知时间

### 风险 2：MCP Server 不可用
**风险：** 用户可能找不到合适的 MCP Server
**缓解措施：**
- 在文档中提供推荐的 MCP Server 列表
- 提供 MCP Server 配置示例
- 如果需求强烈，可考虑未来添加社区维护的 MCP Server

### 权衡 1：便利性 vs 性能
**权衡：** 内置工具更方便使用，但性能较差
**决策：** 选择性能和可扩展性，牺牲一些便利性
**理由：** 长期来看，MCP 架构能提供更好的性能和灵活性

### 权衡 2：功能完整性 vs 核心轻量
**权衡：** 保留 archive 工具使功能更完整，但增加了核心复杂度
**决策：** 选择核心轻量，通过 MCP 扩展功能
**理由：** 符合项目的 MCP 优先架构理念

## Migration Plan

### 步骤 1：准备
- 创建 remove-archive-tools 变更提案
- 审查所有受影响的代码和文档

### 步骤 2：删除代码
- 删除 `niuma/agent/tools/archive.ts`
- 删除 `niuma/__tests__/archive-tools.test.ts`
- 从工具注册表中移除相关代码

### 步骤 3：更新依赖
- 从 package.json 中删除 adm-zip、archiver、tar
- 运行 `pnpm install` 清理依赖

### 步骤 4：更新文档
- 更新 CHANGELOG.md（标记为 BREAKING CHANGE）
- 更新开发计划
- 添加 MCP Server 配置示例

### 步骤 5：验证
- 运行所有测试确保没有遗漏
- 检查构建是否成功

### 回滚策略
如果发现重大问题，可以：
1. 从 git 历史恢复删除的文件
2. 恢复 package.json 中的依赖
3. 运行 `pnpm install` 重新安装依赖
