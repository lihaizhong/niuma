## Decisions

### 1. SessionManager 文件系统集成策略

**决策**: 采用文件系统扫描和 LRU 缓存策略

**理由**:
- 保持与现有缓存机制的一致性
- 文件系统操作是异步的，不会阻塞主线程
- LRU 缓存策略提升性能，同时保持数据一致性

**实施细节**:
- 扫描 `~/.niuma/sessions/` 目录中的所有 `.json` 文件
- 解析每个文件的 SessionKey 提取 channel 和 userId 信息
- 使用 Map 缓存扫描结果，避免重复文件 I/O
- 实现 LRU 缓存淘汰策略，控制内存占用

**替代方案考虑**:
- 使用数据库索引：需要引入新的依赖，增加复杂度
- 实时监听文件变化：需要文件系统监听器，增加复杂度

### 2. 渠道 SDK 集成策略

**决策**: 分阶段实现，优先核心功能

**理由**:
- 降低实施复杂度，便于测试和验证
- 遵循已有渠道（Telegram、Discord）的实现模式
- 确保每个渠道都能独立测试

**实施细节**:
- **阶段 1**: 实现 SDK 集成和事件监听（start 方法）
- **阶段 2**: 实现资源清理（stop 方法）
- **阶段 3**: 实现消息发送（send 方法）

**替代方案考虑**:
- 一次性实现所有功能：增加测试难度，难以定位问题
- 使用通用渠道抽象层：失去特定渠道的特性

### 3. 渠道依赖选择

**决策**: 选择官方或主流 SDK

**理由**:
- 使用官方或主流 SDK 保证稳定性和文档质量
- 部分依赖已存在，减少新增依赖数量
- 选择 TypeScript 原生支持的库

**依赖列表**:
- **QQ**: `oicq` (官方 Node.js SDK)
- **WhatsApp**: `@whiskeysockets/baileys` (WhatsApp Web API)
- **Email**: `nodemailer` (SMTP) + `imapflow` (IMAP) - 已有
- **飞书**: `@larksuiteoapi/node-sdk` (官方 SDK)
- **Slack**: `@slack/bolt` - 已有
- **钉钉**: `dingtalk-sdk` (官方 Node.js SDK)

**替代方案考虑**:
- 使用 HTTP API 直接调用：增加开发工作量，维护困难
- 使用第三方 SDK：可能存在兼容性问题

### 4. Agent ID 配置优化

**决策**: 从 Agent 配置中提取 Agent ID

**理由**:
- 避免硬编码，支持多角色配置
- 保持配置优先级的一致性
- 为未来的多 Agent 协作做准备

**实施细节**:
- 在 `AgentLoop` 构造函数中接收 `agentId` 参数
- 在 `ToolRegistry.setAgentId()` 中使用该参数
- 在 `index.ts` 中从配置管理器获取 `agentId`

**替代方案考虑**:
- 使用环境变量：增加配置复杂度
- 使用全局变量：不符合 TypeScript 最佳实践

### 5. 渠道配置加载优化

**决策**: 实现动态渠道实例化

**理由**:
- 保持与现有配置系统的一致性
- 支持配置热重载（未来扩展）
- 提高代码的可维护性

**实施细节**:
- 在 `ChannelRegistry.loadFromConfig()` 中根据配置创建渠道实例
- 使用工厂模式根据 `channel.type` 创建对应渠道实例
- 支持 `enabled` 字段控制渠道是否启用

**替代方案考虑**:
- 手动注册渠道：增加配置复杂度
- 使用配置文件直接实例化：降低类型安全性

## Risks / Trade-offs

### 风险 1: 渠道 SDK 兼容性问题

**描述**: 新增的渠道 SDK 可能存在 API 兼容性问题或文档不完善的情况。

**缓解策略**:
- 在实施前充分调研 SDK 文档和示例代码
- 优先选择官方或主流维护的 SDK
- 先实现最小可行版本，逐步完善功能
- 编写详细的单元测试和集成测试

### 风险 2: 文件系统性能问题

**描述**: SessionManager 扫描大量会话文件可能导致性能问题。

**缓解策略**:
- 使用 LRU 缓存策略，避免重复文件 I/O
- 限制单次扫描的文件数量
- 实现异步扫描，不阻塞主线程
- 考虑将来迁移到数据库存储

### 风险 3: 配置复杂度增加

**描述**: 新增渠道和配置选项可能导致配置系统复杂度增加。

**缓解策略**:
- 提供默认配置值，降低用户配置门槛
- 编写详细的配置文档和示例
- 实现配置验证和错误提示
- 使用 TypeScript 类型确保配置正确性

### 风险 4: 依赖版本冲突

**描述**: 新增的渠道依赖可能与现有依赖产生版本冲突。

**缓解策略**:
- 在添加依赖前检查版本兼容性
- 使用 `pnpm` 的依赖解析机制
- 在测试环境中充分验证
- 及时更新依赖版本

### 权衡 1: 性能 vs 内存占用

**描述**: SessionManager 使用缓存策略可能增加内存占用。

**权衡**:
- 选择 LRU 缓存策略，平衡性能和内存占用
- 设置合理的缓存大小限制
- 定期清理过期缓存

### 权衡 2: 功能完整性 vs 实施复杂度

**描述**: 完整实现所有渠道功能可能增加实施复杂度。

**权衡**:
- 分阶段实现，优先核心功能
- 遵循已有渠道的实现模式
- 确保每个渠道都能独立测试

### 权衡 3: 依赖数量 vs 功能完整性

**描述**: 新增多个渠道依赖可能增加依赖管理复杂度。

**权衡**:
- 选择官方或主流 SDK，减少维护成本
- 部分依赖已存在，减少新增依赖数量
- 使用 TypeScript 原生支持的库
