### 1.3 AI 陪伴产品用户研究

#### 情感连接形成机制

根据对 612 名中国用户的调研（Frontiers in Psychology, 2025）：

| 变量关系 | 相关系数 | 解读 |
|----------|----------|------|
| 使用频率 → 情感依恋 | β = 0.44 | 高频互动直接促进情感连接 |
| 情感依恋 → 孤独感 | β = -0.32 | 有效缓解孤独 |
| 情感依恋 → 主观幸福感 | β = 0.41 | 显著提升幸福感 |
| 情感依恋 → 自我认同清晰度 | β = 0.51 | **最显著的心理影响** |

#### 关键发现

> **自我认同清晰度是连接情感依恋与现实社交参与的最重要桥梁。**

这意味着 AI 陪伴产品若能帮助用户"更好地认识自己"，将产生最大的积极影响。

#### 设计启示

1. **连续性特征**：保持对话和记忆的连续性
2. **情境记忆**：记住用户的重要信息和偏好
3. **自我表达设计**：鼓励用户探索和表达真实自我

### 2.2 OpenClaw 架构深度分析

#### 核心架构：Hub-and-Spoke

```
                    ┌─────────────────┐
                    │    Gateway      │
                    │  (控制平面)      │
                    └────────┬────────┘
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
   ┌────▼────┐          ┌────▼────┐          ┌────▼────┐
   │ Agent 1 │          │ Agent 2 │          │ Agent 3 │
   │ (main)  │          │ (work)  │          │(family) │
   └─────────┘          └─────────┘          └─────────┘
        │                    │                    │
   ┌────▼────┐          ┌────▼────┐          ┌────▼────┐
   │Workspace│          │Workspace│          │Workspace│
   │+ AgentDir│         │+ AgentDir│         │+ AgentDir│
   │+ Sessions│         │+ Sessions│         │+ Sessions│
   └─────────┘          └─────────┘          └─────────┘
```

#### 一个 Agent 的组成

| 组件 | 说明 |
|------|------|
| **Workspace** | 文件、AGENTS.md/SOUL.md/USER.md、本地笔记、角色规则 |
| **AgentDir** | 认证配置、模型注册、Agent 专属配置 |
| **Session Store** | 聊天历史 + 路由状态 |

#### 路由规则（优先级从高到低）

1. `peer` 匹配（精确 DM/群组/频道 ID）
2. `parentPeer` 匹配（线程继承）
3. `guildId + roles`（Discord 角色路由）
4. `guildId`（Discord）
5. `teamId`（Slack）
6. `accountId` 匹配
7. 频道级别匹配（`accountId: "*"`）
8. 回退到默认 Agent

#### 对 Niuma 的启示

| 特性 | 可借鉴点 |
|------|----------|
| **完全隔离的 Agent** | 不同功能模块可独立演进 |
| **统一的 Gateway 入口** | 单一入口管理所有交互 |
| **灵活的路由绑定** | 根据用户/场景自动路由 |
| **每个 Agent 独立沙箱** | 安全隔离，权限控制 |

### 2.3 记忆系统与上下文管理

#### Anthropic 的上下文工程原则

> **核心原则：在有限的注意力预算内，找到最小的高信号 Token 集合。**

#### 记忆系统架构

```
┌─────────────────────────────────────────────────────────────┐
│                      Context Window                          │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐            │
│  │ System      │ │ Tools       │ │ Message     │            │
│  │ Prompt      │ │ Definition  │ │ History     │            │
│  └─────────────┘ └─────────────┘ └─────────────┘            │
│                                                              │
│  ┌─────────────────────────────────────────────────────────┐│
│  │              External Memory Store                       ││
│  │  • 长期记忆（用户偏好、重要事件）                           ││
│  │  • 项目状态（任务进度、里程碑）                             ││
│  │  • 压缩摘要（对话历史摘要）                                 ││
│  └─────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────┘
```

#### 长时程任务的技术策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **Compaction（压缩）** | 压缩对话历史，保留关键信息 | 需要大量来回对话的任务 |
| **结构化笔记** | Agent 定期将笔记持久化到内存 | 有明确里程碑的迭代开发 |
| **子 Agent 架构** | 主 Agent 协调，子 Agent 处理具体任务 | 复杂研究与分析任务 |
