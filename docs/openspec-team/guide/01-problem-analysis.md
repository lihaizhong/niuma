# 01 - 问题分析

## 三大核心挑战

### 挑战 1: 合并冲突 / 规格冲突 / 规格-代码同步

#### 问题层次

```
冲突类型            当前问题                    影响
────────────────────────────────────────────────────────────
文件冲突            Git merge conflict         手动解决，耗时
规格逻辑冲突        两个 specs 定义冲突接口     系统行为不一致
规格-代码脱钩       改代码不改 specs           文档失去价值
```

#### AI 加速带来的新风险

```
人类开发                     AI 开发
─────────────────────────────────────────────────
速度慢，有时间发现冲突    →   速度快，冲突瞬间产生
逻辑冲突容易察觉          →   可能生成冲突代码而不自知
人工 review 发现不一致    →   需要自动化检测
```

#### 具体冲突场景

**场景 1：AI 并行生成冲突代码**

```
开发者 A: "实现用户登录 API"           开发者 B: "实现用户注册 API"
      │                                    │
      ▼                                    ▼
AI 读取 specs/auth/user.md          AI 读取 specs/auth/user.md
      │                                    │
      ▼                                    ▼
生成: POST /api/users/login         生成: POST /api/users/register
      │                                    │
      ▼                                    ▼
同时使用 User 模型，但:               修改了 User 模型添加字段
  - A 版本: User 有 password_hash     - B 版本: User 有 password + salt
      │                                    │
      ▼                                    ▼
合并时: 两个版本的 User 模型冲突！
  - 数据库 schema 不一致
  - API 返回格式不同
  - 测试全部失败
```

**AI 不会自动检测的问题**：

1. **语义冲突** - 两个 spec 定义了同名但不同含义的接口
2. **隐式依赖** - 变更 A 依赖变更 B 的规格，但 B 还没完成
3. **范围蔓延** - AI 生成的代码超出了 spec 定义的范围

### 挑战 2: 文档-代码不同步

#### 根本原因

```
传统文档问题                    OpenSpec 现状
─────────────────────────────────────────────────
"文档是写给人看的"     →       specs/ 目录可能没人看
"改代码容易，改文档难"  →       specs/ 和代码分离维护
"不知道文档在哪里"      →       没有强制关联机制
```

#### 文档-代码脱钩的典型表现

```
                    理想中的状态

    specs/Feature-A/spec.md        src/Feature-A/impl.ts
            │                               │
            │         同步                    │
            └───────────────┬───────────────┘
                            │
                            ▼
                    ┌───────────────┐
                    │   文档=代码    │
                    └───────────────┘


                    现实中的状态

    specs/Feature-A/spec.md        src/Feature-A/impl.ts
            │                               │
            │         脱钩                    │
            └───────────────┬───────────────┘
                            │
                            ▼
            ┌───────────────────────────────┐
            │  文档描述 v1，代码实现 v2        │
            │  specs 说"需要用户名"，          │
            │  代码里已经改成"邮箱或手机号"     │
            └───────────────────────────────┘
```

### 挑战 3: 跨团队协作

#### 当前问题

| 维度         | 现状         | 问题                     |
| ------------ | ------------ | ------------------------ |
| **前后端**   | Swagger 文档 | 各自维护，变更靠口头约定 |
| **微服务间** | 独立文档     | 没有统一的契约管理       |
| **变更通知** | 口头约定     | 容易遗漏，无法追溯       |

#### 理想协作模式

```
              specs/
           ┌──────────────┐
           │  Interface   │  ◀──── 前后端共同维护的契约
           │  Contract    │
           └──────┬───────┘
                  │
       ┌──────────┼──────────┐
       │          │          │
       ▼          ▼          ▼
   前端实现    后端实现     契约测试
   (Consumer) (Provider)   (验证)
```

## 规模带来的复杂度跳跃

```
┌─────────────────────────────────────────────────────────┐
│  10人团队                    80+人团队                    │
├─────────────────────────────────────────────────────────┤
│  口头沟通即可                 需要显式契约                 │
│  一人了解全局                 需要文档传播知识              │
│  单服务/少量服务              多服务网状依赖               │
│  冲突少                       冲突频繁                     │
│  代码即文档                   代码≠文档（必须分离维护）      │
└─────────────────────────────────────────────────────────┘
```

## OpenSpec 当前缺失的能力

| 能力         | 当前状态 | 80+团队风险            |
| ------------ | -------- | ---------------------- |
| **版本控制** | ❌ 无    | 制品冲突无法追踪       |
| **依赖管理** | ❌ 无    | 变更间依赖不明         |
| **冲突检测** | ❌ 无    | 多人修改同一制品会覆盖 |
| **并发控制** | ❌ 无    | 无法锁定正在处理的变更 |
| **权限控制** | ❌ 无    | 任何人都可归档任何变更 |
| **变更排队** | ❌ 无    | 资源竞争无协调机制     |
| **状态广播** | ❌ 无    | 团队成员不知道他人进展 |
| **审计日志** | ❌ 无    | 变更历史无法追溯       |
